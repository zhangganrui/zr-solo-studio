# 编排细则

总指挥在对话中按需查阅。SKILL.md 是主干流程，这里是三道缝的展开做法 + 检查点话术 + 边界处理。

---

## project-slug 命名

从想法提炼一个短横线英文名，第一两轮内静默定下，全程复用。
- 例："AI 内容内化器" → `ai-internalizer`；"创作者提现" → `creator-payout`。
- 只用小写字母 + 短横线，2-3 个词以内。

---

## 缝一 · 美学方向（决策表）

taste-pm 给的是结构 + 字节系 Semi/Arco 组件清单（中性偏商务）；awesome 需要一个品牌名才能出视觉。你的任务：根据**产品类型 + 目标用户 + 核心情绪**，从 awesome 的 113 个品牌里推荐一个参照，给理由 + 次选。

参考映射（不是死规则，是判断起点）：

| 产品气质 / 情绪 | 推荐参照品牌 | 关键识别符 |
|---|---|---|
| 安静、专注、极客、效率 | Linear / Warp / Cursor | 暗色画布、极简、克制、单一强调色 |
| 柔和、亲切、有人味 | Notion / Slack / Airbnb | 浅色、暖调、彩色卡片、圆角 |
| 商务、清爽、可信 | 飞书 / Stripe / 腾讯云 | 浅色、网格、克制蓝、清晰层级 |
| 科技、前沿、AI 感 | DeepSeek / Claude / Vercel | 极简、留白、单色或无强调色 |
| 高端、质感、有张力 | 小米汽车 / Ferrari / Apple | 大图、强对比、留白、产品为主角 |
| 内容、年轻、活泼 | 哔哩哔哩 / 抖音 / Spotify | 高饱和、圆角、胶囊按钮 |

输出 `02-aesthetic.md`，三件事即可：
1. **参照品牌**：选定一个（+ 一句话为什么它配这个产品的情绪）。
2. **气质关键词**：3-5 个（如：暗色 / 克制 / 专注 / 无多余强调色）。
3. **强调色策略**：用不用、用什么（照搬该品牌的签名色判断）。

**检查点3 话术**（这是品味判断，必须让人选）：
> "你这个〔产品〕，用户要的是〔核心情绪〕。我推荐走 **〔品牌A〕** 那种气质——〔一句理由〕。次选 〔品牌B〕，如果你想要更〔某倾向〕的话。用哪个？"

## 缝二 · 设计落地（tokens 抽取清单）

awesome 产出的是单文件 HTML mockup，design tokens 写在 `:root` 里。把它变成 cto 能用的设计输入：

从 mockup 里抽出并整理成 `02b-design-tokens.md`：
- [ ] **色**：主色 / 背景 / 文字 / 边框 / 强调色 / 状态色 → 整理成命名变量。
- [ ] **字**：字体族（含中英文 fallback）、字号阶梯、字重、字间距。
- [ ] **形**：圆角阶梯、间距单位（通常 8px 倍数）、阴影策略。
- [ ] **组件**：按钮（胶囊/直角）、输入框、卡片、标签——形态与状态。
- [ ] **主题落地形式**：根据将选技术栈，给成 CSS variables 或 Tailwind `theme.extend`。
- [ ] **页面→组件映射**：把 taste-pm 的页面骨架拆成组件树，对齐 tokens。

目标：cto 拿到这份就能直接照着实现，不用再回头啃 HTML。

## 缝三 · 上线作品化（清单）

cto 默认停在设计/交付给 coding agent；本流水线必须把它跑到"真上线 + 成作品"。写 `SHIP.md`：

**1. 验证能跑**
- [ ] 真正启动它（参照 verify / run 的思路），不是只看代码。
- [ ] 核心流程能走通（real-demand 里定义的"核心动作"）。
- [ ] 跑不起来 → 回第④步让 cto 修，不要带病上线。

**2. 部署上线**（按形态选最简通道）
- 纯 HTML / 静态前端 → 可用 **lark-apps** 部署成公网可访问链接（妙搭）。
- 前端应用 → 最简静态托管。
- 含后端 → 给出最简部署路径，标注需要用户配置的部分（域名、密钥等）。

**3. 作品化**
- [ ] README：解决什么问题 / 核心方法 / 安装用法 / 截图 / 致谢。
- [ ] LICENSE（默认 MIT，除非用户另定）。
- [ ] 截图或 demo 链接。
- [ ] （用户同意后）gh 发布 GitHub，仓库名遵循 **`zr-` 前缀**规范；skill 类产物的 skill 目录名保持无前缀（触发标识，越短越好）。

**对外发布前必须用户确认**——公开即对外，可能被搜索/索引。

## 从中途切入

用户不一定从头开始。常见入口：
- 已有真需求结论 → 从第②步 taste-pm 切入。
- 已有产品结构 / 设计稿 → 从缝一或第③步切入。
- 已有代码、要继续做 → 直接走第④步 cto 的 brownfield 模式 + 缝三。

切入前先一句话确认"我们从〔某步〕开始，对吗"，并尽量把已有材料归位到 `<project-slug>/` 对应文件，保持产物契约完整。

## 检查点总表

| 检查点 | 位置 | 停下来给用户看什么 | 为什么停在这 |
|---|---|---|---|
| 检查点1 | ①之后 | 真需求判决（做/不做/改）| 决定要不要继续，最省钱 |
| 检查点2 | ②之后 | 产品结构（IA + 核心页面）| 结构错则全错，还没画图 |
| 检查点3 | 缝一之后 | 美学方向（参照品牌 + 气质）| 品味判断，且还没出视觉 |

`--auto` 模式下三个都不停，按推荐走，最后统一汇报。
半自动模式下：检查点必停；但检查点之间不要反复问"要继续吗"——用户点头方向后就一路推进到下一个检查点。
